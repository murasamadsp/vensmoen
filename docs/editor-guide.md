# Vensmoen mottak - інструкція для редактора

Це коротка робоча інструкція для працівників, які редагують сайт через
Pages CMS.

## 1. Де редагувати

Відкрийте `https://app.pagescms.org`, увійдіть через GitHub і відкрийте
репозиторій `murasamadsp/vensmoen`.

У CMS зазвичай має бути два розділи:

- `Norsk innhold` - увесь основний текст сайту норвезькою.
- `Driftsdata` - черговий телефон, Mental Helse, посилання на карти,
  транспортні застосунки і дата останньої перевірки інформації про сортування.

Інші мови напряму не редагуються. Коли ви зберігаєте норвезький текст, GitHub
автоматично перекладає зміни іншими мовами.

## 2. Як оновити текст

1. Відкрийте `Norsk innhold`.
2. Знайдіть потрібну сторінку або тип відходів.
3. Змініть текст.
4. Натисніть Save.
5. Дочекайтеся завершення GitHub Actions.

Якщо workflow зелений, сайт опублікується автоматично. Якщо workflow червоний,
нова версія не опублікована. У такому разі треба звернутися до технічного
відповідального.

## 3. Важливі правила

- Не додавайте інформацію, яку не підтвердила адміністрація.
- Не редагуйте інші мовні JSON-файли напряму.
- Не видаляйте поля, якщо не впевнені, для чого вони потрібні.
- Пишіть коротко. Багато мешканців читають сайт з телефона.
- Для списків починайте кожен рядок з `- `.
- Не вставляйте HTML у текстові поля.

Приклад списку:

```text
- Перший пункт.
- Другий пункт.
- Третій пункт.
```

## 4. Driftsdata

У `Driftsdata` можна змінювати:

- черговий телефон;
- телефон Mental Helse;
- посилання Google Maps / Apple Maps;
- посилання Entur / Vy / Reis Nordland;
- дату останньої перевірки інформації про сортування.

Телефон для посилання треба писати без пробілів, наприклад `+4748201948`.
Телефон, який показується на сайті, може мати пробіли, наприклад
`+47 482 01 948`.

Дата пишеться у форматі `YYYY-MM-DD`, наприклад `2026-05-18`.

## 5. Що може піти не так

### Сайт не оновився після Save

Найчастіше це означає, що GitHub Actions ще працює або workflow впав.
Перевірте вкладку Actions у GitHub. Якщо прогін червоний, стара версія сайту
залишається в продакшені.

### Переклад не спрацював

Можливі причини:

- немає API-ключа для вибраного провайдера;
- закінчилися кредити або billing у провайдера;
- модель видалена або більше не підтримується;
- провайдер тимчасово недоступний;
- GitHub Secret або Variable названі неправильно.

Налаштування перекладу:

- `TRANSLATE_PROVIDER`: `anthropic`, `deepseek` або `google`;
- `TRANSLATE_MODEL`: основна модель;
- `TRANSLATE_FALLBACK_PROVIDER`: резервний провайдер;
- `TRANSLATE_FALLBACK_MODEL`: резервна модель;
- `TRANSLATE_TIMEOUT_MS`: timeout API-запиту;
- `TRANSLATE_RETRIES`: повтори для 408/429/5xx.

Secrets:

- `ANTHROPIC_API_KEY`;
- `DEEPSEEK_API_KEY`;
- `GOOGLE_TRANSLATE_API_KEY`.

Має бути активний принаймні ключ того провайдера, який вибраний у
`TRANSLATE_PROVIDER`. Якщо є fallback-провайдер, його ключ також має бути
активний.

### GitHub пише, що модель недоступна

Це означає, що модель у `TRANSLATE_MODEL` застаріла, видалена з API або
недоступна для акаунта. Треба замінити модель у GitHub Variables або увімкнути
fallback через `TRANSLATE_FALLBACK_PROVIDER` / `TRANSLATE_FALLBACK_MODEL`.

Для ранньої перевірки є workflow `Translation health check`. Його можна
запустити вручну в GitHub Actions.

### CMS показує помилку при збереженні

Перевірте поля:

- e-mail має виглядати як `name@example.no`;
- URL має починатися з `https://`;
- дата має бути у форматі `YYYY-MM-DD`;
- телефон для посилання має бути без пробілів.

### Зник текст або поле

Не створюйте нові поля вручну в JSON. Pages CMS зберігає файл за схемою
`.pages.yml`; ключі, яких немає у схемі, можуть бути видалені при збереженні.
Нові поля має додавати технічний відповідальний.

### Сторінка друку не вміщується на A4

Причина зазвичай у занадто довгому тексті в сортуванні. Треба або скоротити
текст, або попросити технічного відповідального підлаштувати layout друку.

### Посилання відкривається неправильно

Перевірте, що URL починається з `https://` і веде на стабільну сторінку.
Для карт краще використовувати короткі, перевірені посилання Google Maps або
Apple Maps.

## 6. Після великих змін

Перевірте сайт на телефоні й на комп'ютері. Особливо:

- головну сторінку;
- `Om mottaket`;
- `Husregler`;
- `Praktisk guide`;
- `Kildesortering`;
- друковану сторінку сортування.

Якщо щось виглядає зламаним, не робіть повторні випадкові правки. Спочатку
перевірте GitHub Actions і зверніться до технічного відповідального.
