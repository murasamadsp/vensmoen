# AGENTS.md — Vensmoen mottak (інформаційний сайт центру)

Інструкції для AI-агентів і людей у проєкті **vensmoen** — багатомовному
статичному інформаційному сайті для мешканців центру прийому Vensmoen
(Рокланд, комуна Салтдал). Розділи: «Про центр», «Правила проживання»,
«Практичний гід» та повний гід із сортування відходів (`/waste/`).

## Огляд проєкту

- **Мета:** простий, охайний і придатний для друку сайт із постійною
  інформацією для мешканців центру.
- **Аудиторія:** біженці, іммігранти та нові мешканці Норвегії.
- **Технології:** [Astro](https://astro.build) (статичний вивід), без
  UI-фреймворків (vanilla JS/CSS), Biome для лінтингу та форматування.
- **Контент:** сортування перевірено за національним стандартом **sortere.no**.

## Залізні правила

1. **Не вигадуємо фактів.** Контент сайту — лише підтверджена інформація від
   адміністрації центру. Немає інформації → чесна заглушка
   (`site.comingSoon` або блок без `body`).
2. **`nb.json` — структурний еталон.** Набір ключів і форма масивів звіряються
   з норвезьким файлом; зміст можна редагувати в будь-якій мові через CMS.
3. **Жодного runtime-JS без потреби** (виняток — квіз).
4. **Дизайн-токени** з `src/styles/tokens.css`, системні шрифти.
5. **Однакова висота карток у всіх мовах:** довжина текстів різна, тому зони
   карток резервують фіксовану кількість рядків (див. `.emg__label`,
   `.contact-card__*`).
6. **Коментарі в коді — українською.**
7. **Друк:** зміни макета перевіряємо в `/[locale]/waste/print/` — все має
   вміщатися на один A4.

## Архітектура та важливі файли

### Інтернаціоналізація (i18n)

Ручне рішення з маршрутизацією через `/[locale]/`.

- `src/i18n/locales.ts` — список мов (код, назва, напрям тексту/dir).
- `src/i18n/strings.ts` — типізований лоадер словників.
- `src/i18n/<код>.json` — файли перекладів; **`nb.json` — структурний
  еталон**.
- `scripts/validate-i18n.mjs` — перевірка синхронності ключів з `nb.json`.

### Дані

- `src/data/fractions.ts` — мовонезалежні метадані фракцій (`binColor`,
  `sources`, `lastVerified`).
- `src/data/emergency.ts` — національні екстрені номери + черговий телефон
  центру (`dutyPhone`).
- `src/data/location.ts` — посилання на карти; `src/data/transport.ts` —
  застосунки з розкладом транспорту.

### Сторінки та компоненти

- `src/pages/[locale]/index.astro` — головна: хаб із секційними картками,
  черговий телефон, екстрені номери.
- `src/pages/[locale]/about/` — про центр: офіс, контакти, карти, транспорт.
- `src/pages/[locale]/rules/` — правила проживання.
- `src/pages/[locale]/guide/` — практичний гід (контент частково заглушки).
- `src/pages/[locale]/waste/` — гід із сортування; `waste/quiz/` — квіз;
  `waste/print/` — A4-аркуш із QR-кодом.
- `src/components/InfoPage.astro` — спільний шаблон інфосторінок.
- `src/components/FractionCard.astro`, `GuideBody.astro` — спільні для веба
  та друку (DRY).

## Команди

| Команда | Опис |
| :--- | :--- |
| `npm run dev` | Локальний сервер розробки. |
| `npm run build` | Збірка статичного сайту в `dist/`. |
| `npm run check` | Валідація i18n + перевірка типів + лінтинг (обов'язково перед push). |
| `npm run validate:i18n` | Перевірка синхронності перекладів з `nb.json`. |
| `npm run lint` / `npm run lint:write` | Biome: перевірка / автовиправлення. |
| `npm run validate:cms` | Перевіряє, що Pages CMS покриває всі ключі `nb.json` і експонує всі мови. |
| `npm run validate:translate-env` | Перевіряє локальний/CI env для авто-перекладу. |
| `npm run translate -- --source <код> --paths a.b,c [--dry]` | Ручний переклад заданих полів з мови-джерела (провайдер/модель із `.env`). |
| `npm run translate -- --fill-todo [--dry]` | Заповнити всі значення `[TODO] ...` перекладом з `nb`. |
| `npm run translate:health` | Малий API health-check вибраного провайдера/моделі без запису файлів. |
| `npm run sync:i18n` | Синхронізувати структуру ключів усіх мов із `nb.json`. |

## Як додати нову мову

1. `cp src/i18n/en.json src/i18n/<код>.json` і перекладіть значення
   (ключі не чіпати).
2. Імпортуйте файл у `src/i18n/strings.ts` і додайте в `dictionaries`.
3. Додайте рядок у `src/i18n/locales.ts` (`code`, `label`, `dir`,
   `htmlLang`); для RTL-скриптів — `dir: 'rtl'`.
4. `npm run check && npm run build` — валідатор зловить пропущені ключі,
   маршрути `/<код>/` створяться автоматично.
5. Додайте запис мови в `.pages.yml` (секція `content:`) з `fields: *fields`
   та `format: json`, інакше мова не з'явиться в CMS.

## Оновлення контенту сортування

Після звірки фракції з [sortere.no](https://sortere.no/) оновіть текст у
потрібному `src/i18n/<код>.json` (`fractions`) **і** дату
`wasteLastVerified` у `src/data/site-settings.json`. Інші мови може оновити
авто-переклад.

## CMS і авто-переклад (Pages CMS + env-провайдер)

Контент редагується **нетехнічним** працівником лише через веб-CMS
[Pages CMS](https://pagescms.org) (`app.pagescms.org`, вхід через GitHub) —
**без термінала**. Конфіг — `.pages.yml`; CMS редагує всі
`src/i18n/<код>.json` (один запис на мову) і `src/data/site-settings.json`
(`Driftsdata`). Трекер доробок — `docs/CMS-TRACK.md`.

### ⚠️ КРИТИЧНО: схема `.pages.yml` мусить покривати ВСІ ключі

Pages CMS при збереженні **видаляє** будь-які ключі JSON, яких немає в схемі
`.pages.yml`. Тому:

- Додав новий ключ у контент (`nb.json`) → **обов'язково** додай його і в
  `.pages.yml` (у спільний YAML-анкер `&fields`), інакше CMS зріже його при
  першому ж збереженні мови.
- `validate:cms` ловить непокриті ключі до деплою; `validate:i18n` ловить
  десинхрон мов і `[TODO]`.
- Перевірка повноти: порівняй усі листові шляхи `nb.json` зі шляхами полів у
  `.pages.yml` — розбіжностей «у JSON, але не в схемі» має бути нуль.
- Системні рядки `ui` приховані в CMS через `hidden: true` (лишаються у файлах,
  редактор їх не бачить).
- Не дробити `src/i18n/nb.json` на кілька часткових Pages CMS `file`-записів
  без live-тесту: Pages CMS зберігає файл за схемою, і ключі поза схемою можуть
  бути зрізані.

### Авто-переклад (CI)

Вшитий у `.github/workflows/deploy.yml` (крок перед build),
скрипт `scripts/auto-translate.mjs`. Провайдер і модель задаються через
`TRANSLATE_PROVIDER` / `TRANSLATE_MODEL` у GitHub Variables; ключі — через
GitHub Secrets (`ANTHROPIC_API_KEY`, `DEEPSEEK_API_KEY`,
`GOOGLE_TRANSLATE_API_KEY`). Локальний приклад конфігу — `.env.example`;
реальний `.env` не комітити. Працює сам на кожен Save у CMS:

- **Розумний:** CI дивиться git-diff; мова, яку змінили в CMS, стає джерелом
  для цього поля, і зміна перекладається на решту мов. Якщо одночасно змінено
  кілька мов для одного поля, пріоритет має `nb`.
- **Без нескінченного циклу:** бот комітить переклади як автор
  `vensmoen-translate-bot` через `GITHUB_TOKEN`, а такі push НЕ тригерять новий
  workflow. 1 правка людини → 1 раунд перекладу → стоп.
- **Без вічного десинку:** база diff = останній **успішний** комміт бота (а не
  попередній push). Якщо прогін упав, база лишається, і наступний прогін
  наздоганяє всі накопичені зміни.
- **Fail-closed:** немає ключа / кредитів / модель deprecated / помилка API →
  deploy падає. Новий контент не публікується, поки всі мови не синхронні.
- **Не палить даремно:** якщо `src/i18n` не змінювався (тільки CSS/доки) —
  API не запускається взагалі.
- **Надійний вивід:** відповідь провайдера перевіряється на точний набір ключів,
  string-значення і непорожні тексти; системний стиль-гайд НЕ перекладає власні
  назви (Vensmoen, Saltdal, Røkland, UDI, NAV, Mental Helse, sortere.no).
- **Health-check:** `.github/workflows/translate-health.yml` раз на тиждень і
  вручну перевіряє, що вибрані provider/model ще живі. Якщо модель видалили з
  API, помилка пояснює, які GitHub Variables/Secrets оновити.

### Структурний референс

Структурний еталон — `nb.json`: набір ключів і форма масивів звіряються з ним.
Змістовно авто-переклад бере джерелом мову, яку відредагували в CMS.
`--fill-todo` бере текст із маркера `[TODO] <nb>`.

### Деплой і репозиторій

Push у `main` = деплой на GitHub Pages. Origin форка вказує на
`github.com/murasamadsp/vensmoen`. Ручні прогони перекладу роби з `--dry`
спочатку, потім звір `git diff` (відкат — `git restore src/i18n/`).
